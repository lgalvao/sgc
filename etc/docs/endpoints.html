🔐 Autenticação — /api/usuarios

Controller: LoginController. Não requer autenticação prévia. Gerencia o fluxo de dois passos: credenciais → seleção de perfil.

POST /api/usuarios/login Iniciar sessão
Finalidade

Autentica o usuário (credenciais verificadas contra o LDAP/base). Se o usuário possui apenas um perfil/unidade, já conclui o login e retorna o JWT. Se possui múltiplos perfis, retorna lista de opções e um cookie temporário SGC_PRE_AUTH (5 min) para o passo 2.

Acesso
Público (sem auth)
Body (JSON)
{ "tituloEleitoral": "12345678901", "senha": "••••••" }
Resposta
{ "autenticado": true, "requerSelecaoPerfil": false, "perfisUnidades": [...], "sessao": { "tituloEleitoral": "...", "nome": "...", "perfil": "ADMIN", "unidadeCodigo": 1, "permissoes": {...} } }
Uso no frontend
usuarioService.login() → store de autenticação Pinia → redireciona para painel ou tela de seleção de perfil.
Notas de implementação
  • Rate limiting via LimitadorTentativasLogin por IP
  • Cookies são HttpOnly; Secure configurável via aplicacao.cookies.secure
  • Cookie JWT tem maxAge=86400 (24h); PRE_AUTH tem maxAge=300 (5min)
  • Sanitização de IP contra log injection (CWE-117)
Análise
✅ Endpoint bem projetado. O fluxo de dois passos para múltiplos perfis é elegante. Considera-se adequado.
💡 Melhoria potencial: o campo requerSelecaoPerfil na resposta é redundante com sessao == null. Poderia ser simplificado.
POST /api/usuarios/entrar Selecionar perfil (passo 2)
Finalidade

Passo 2 do login. Chamado apenas quando o usuário tem múltiplos perfis/unidades. Requer cookie SGC_PRE_AUTH válido. Emite o JWT final e apaga o cookie temporário.

Acesso
Requer cookie SGC_PRE_AUTH
Body (JSON)
{ "perfil": "CHEFE", "unidadeCodigo": 42 }
Uso no frontend
usuarioService.entrar() → mesmo store de auth.
Análise
✅ Necessário e correto. Sem problemas identificados.
POST /api/usuarios/logout Encerrar sessão
Finalidade

Remove os cookies jwtToken e SGC_PRE_AUTH (maxAge=0). Sem blacklist de tokens — o JWT continua tecnicamente válido até expirar.

Acesso
Público (sem auth)
Análise
⚠️ Não há invalidação server-side do JWT. Em cenários de segurança elevada (comprometimento de sessão), o token continua válido até expirar (24h). Melhoria: implementar blacklist de tokens (Redis) ou usar tokens de curta duração + refresh token.

🏠 Painel — /api/painel

Controller: PainelController. Dashboard principal do sistema. Todos os endpoints usam o contexto do JWT para filtrar dados.

GET /api/painel/bootstrap Dados iniciais do painel (BFF)
Finalidade

Endpoint BFF (Backend-for-Frontend) que retorna em uma única chamada tudo que o painel precisa na carga inicial: processos relevantes + alertas não lidos + contadores. Reduz drasticamente o número de roundtrips na abertura da tela.

Acesso
Todos os perfis autenticados
Uso no frontend
painelService.obterBootstrap() → store Pinia usePainelStore na montagem da view do painel.
Análise
✅ Excelente padrão BFF. Substitui múltiplas chamadas paralelas. Mantê-lo é prioritário para performance.
GET /api/painel/processos Lista paginada de processos do painel
Finalidade

Lista paginada dos processos visíveis ao usuário autenticado (baseado no perfil e unidade do JWT). ADMIN vê todos; perfis menores veem apenas processos de sua unidade ou hierarquia.

Acesso
Todos os perfis autenticados
Query Params
ParamTipoDescrição
pageintNúmero da página (0-based)
sizeintTamanho (padrão: 20)
sortstringCampo de ordenação
Uso no frontend
painelService.listarProcessos() → paginação no componente do painel.
Análise
💡 O endpoint /bootstrap já retorna os processos iniciais. Este endpoint é usado para paginação subsequente — faz sentido existir. Avaliar se o tamanho padrão de 20 é suficiente para evitar paginação na maioria dos casos.
GET /api/painel/alertas Lista paginada de alertas do painel
Finalidade

Alertas do usuário conforme CDU-02: alertas pessoais do usuário ou de sua unidade ativa (sem recursividade na hierarquia).

Acesso
Todos os perfis autenticados
Uso no frontend
painelService.listarAlertas()
POST /api/painel/alertas/marcar-lidos Marcar alertas como lidos
Finalidade

Marca uma lista de alertas como lidos para o usuário autenticado. Chamado assincronamente pelo frontend após exibir a lista.

Acesso
Todos os perfis autenticados
Body (JSON)
[1, 2, 3, 42]
Uso no frontend
painelService.marcarAlertasLidos() — fire-and-forget.
Análise
✅ Endpoint consolidado para marcação em lote de alertas lidos no painel.

📋 Processos — /api/processos

Controllers: ProcessoController (todos os ambientes) + ProcessoExclusaoController (profile=hom). Gerencia o ciclo de vida dos processos de mapeamento, revisão e diagnóstico.

POST /api/processos Criar processo
Finalidade

Cria um novo processo (MAPEAMENTO, REVISÃO ou DIAGNÓSTICO) no estado CRIADO. O processo fica visível mas não ativo até ser iniciado.

Acesso
ADMIN
Body (JSON)
{ "descricao": "Mapeamento 2025", "tipo": "MAPEAMENTO", "dataLimite": "2025-12-31" }
Resposta
201 Created com ProcessoResumoDto e header Location.
Uso no frontend
processoService.criarProcesso() → tela de criação de processo (só visível para ADMIN).
GET /api/processos/{codigo} Obter processo por código
Acesso
ADMIN ou usuário com acesso ao processo via hierarquia.
Uso no frontend
Usado pontualmente em telas de detalhe.
Análise
💡 Este endpoint retorna apenas o resumo (ProcessoResumoDto). Para o detalhe completo, o frontend usa /detalhes ou /contexto-completo. Considerar se este endpoint ainda tem uso específico ou pode ser unificado.
GET /api/processos/{codigo}/detalhes Detalhes do processo
Finalidade

Retorna o ProcessoDetalheDto com todas as unidades participantes e seus subprocessos. Usado em telas de acompanhamento.

Acesso
ADMIN ou acesso via hierarquia.
Uso no frontend
processoService.obterDetalhesProcesso()
GET /api/processos/{codigo}/contexto-completo Contexto completo do processo (BFF)
Finalidade

Endpoint BFF — retorna tudo que a view de detalhe de processo precisa de uma vez, incluindo permissões calculadas para cada subprocesso. Chamado na abertura da página de processo.

Acesso
ADMIN ou acesso via hierarquia.
Uso no frontend
processoService.buscarContextoCompleto()
Análise
✅ Padrão BFF correto. Diferente de /detalhes (sem permissões calculadas): este inclui as permissões de UI para cada subprocesso, evitando N+1 requests no frontend.
POST /api/processos/{codigo}/iniciar Iniciar processo
Finalidade

Transiciona o processo de CRIADO para EM_ANDAMENTO. Recebe a lista de unidades participantes e cria os subprocessos correspondentes. Dispara notificações para os responsáveis.

🔀 Subprocessos — /api/subprocessos

Controller: SubprocessoController. Este é o controller mais complexo do sistema, concentrando ~50 endpoints que cobrem todo o fluxo de cadastro de atividades e validação de mapas de competências.

📌 Padrão de permissões: A maioria dos endpoints usa @PreAuthorize("hasPermission(..., 'Subprocesso', 'ACAO')"), delegando ao SgcPermissionEvaluator a lógica de controle baseada na hierarquia organizacional e localização atual do subprocesso.

Consultas e contexto

GET /api/subprocessos/{codSubprocesso}/contexto-edicao Contexto de edição (BFF)
Finalidade

Endpoint BFF que retorna tudo que a tela de edição de cadastro de atividades precisa: detalhes do subprocesso, atividades atuais, permissões de UI calculadas. Evita múltiplas chamadas da view.

Acesso
VISUALIZAR_SUBPROCESSO
Uso no frontend
subprocessoService.buscarContextoEdicao()
GET /api/subprocessos/contexto-edicao/buscar Contexto de edição por processo+unidade
Finalidade

Mesmos dados que o endpoint acima, mas buscado pelo par codProcesso + siglaUnidade em vez do código do subprocesso. Útil quando o frontend só conhece o processo e a unidade (rota de URL).

Acesso

🔍 Análise e Recomendações

Pontos positivos

  • ✅ Uso consistente do padrão BFF (/bootstrap, /contexto-completo, /contexto-edicao) reduz N+1 requests
  • ✅ Perfis Spring (hom, e2e) isolam endpoints perigosos de forma robusta
  • ✅ AtividadeOperacaoResponse retorna estado atualizado eliminando re-fetch
  • ✅ Convenção POST com sufixo semântico (/excluir, /iniciar, etc.) bem documentada e consistente
  • ✅ Permissões granulares via SgcPermissionEvaluator — separação clara de concerns
  • ✅ SSE para invalidação de cache em tempo real é elegante e sem polling
  • ✅ Sanitização de inputs (HTML escape, log injection) em pontos corretos
  • ✅ Relatórios seguem padrão JSON + PDF também para /unidades-sem-mapas-vigentes

Problemas e melhorias identificadas

#SeveridadeEndpoint(s)ProblemaRecomendação
1 Média /processos/{id}/unidades-importacao Restrito a CHEFE e ADMIN; GESTOR permanece bloqueado Revisar se GESTOR deve ter acesso conforme regra de negócio e casos de uso.
2 Baixa /mapas Sem paginação, pode retornar volume grande Adicionar paginação ou pelo menos filtros básicos.
3 Baixa /processos/finalizados
/processos/para-importacao		 Dois endpoints similares, diferem apenas por filtro de elegibilidade de importação Considerar unificar com parâmetro ?elegívelImportacao=true — mas a separação atual tem clareza de intenção.
4 Baixa /unidades/{id}/mapa-vigente Retorna apenas booleano quando /mapa-vigente/referencia já contém essa informação Avaliar se há consumidores que usam somente o booleano; se não, remover.

Oportunidades de evolução arquitetural

  1. WebSockets / SSE expandido: O mecanismo SSE existente pode ser expandido para notificações em tempo real de mudanças de estado de subprocesso, reduzindo a necessidade de polling manual.
  2. Endpoint de contexto global: Um endpoint único de "bootstrap da aplicação" que retorne dados organizacionais (unidades, configurações) junto com os dados do usuário, reduzindo as 3-4 chamadas iniciais.
  3. Paginação consistente: Padronizar paginação em todos os endpoints de listagem que podem crescer (mapas, notificações, feedbacks).
  4. Versionamento de API: Sem versionamento atual (/v1/). Considerar para mudanças breaking futuras.
  5. Consolidação de endpoints de bloco: Os endpoints /bloco duplicam lógica de permissões dos endpoints individuais. Avaliar se um padrão genérico de operação em lote seria mais manutenível.
